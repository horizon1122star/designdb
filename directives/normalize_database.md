# Directive: Normalize Database (3NF)

> **Purpose:** Validate the LLM-generated JSON schema for anomalies and automatically decompose it into Third Normal Form (3NF) to ensure data integrity and eliminate redundancy.

---

## Overview

This directive outlines the process of validating and normalizing the database extracted during the `analyze_requirements` phase. Normalization algorithms here are **deterministic** rather than probabilistic, operating on mathematical functional dependencies and structural heuristics to achieve 1NF, 2NF, and ultimately 3NF.

---

## Inputs

**Required:**
- `parsedSchema` (JSON): The intermediate schema structure generated by the LLM.
- `functionalDependencies` (JSON/Array): Mappings of input dependencies (either inferred in the previous step or explicitly defined).

**Optional:**
- `strictMode` (boolean): If true, enforces strict BCNF. If false, allows common denormalizations (like keeping `city`/`state` combined for small projects).
- `autoDecompose` (boolean): If true, automatically refactors the schema. If false, generates a normalization report for user review.

---

## Tools to Use

**Primary:**
- `execution/normalize_schema.ts`
  - Runs deterministic TS algorithms (Candidate Key detection, Functional Dependency analysis).
  - Validates 1NF, 2NF, 3NF compliance.
  - Generates decomposed entities and updates relationships.

**Helper Algorithms (Internal to TS):**
- *Dependency Graph Builder*: To detect transitive relationships.
- *Lossless Join Verifier*: To ensure decomposed tables can be accurately rejoined.

---

## Processing Logic & 3NF Engine

### 1. First Normal Form (1NF) Validation
**Goal:** Ensure Domain Atomicity and eliminate repeating groups.
- **Algorithm:** Scan all attribute names and data types.
  - Look for comma-separated intent (e.g., `VARCHAR` attributes named `phone_numbers`, `tags`).
  - Look for repeating column patterns (`phone1`, `phone2`, `phone3`).
- **Resolution:** If found, extract into a new entity (e.g., `user_phone`) with a Many-to-One relationship to the parent entity.

### 2. Second Normal Form (2NF) Validation
**Goal:** Eliminate Partial Dependencies.
- **Condition:** Only applies to entities with **Composite Primary Keys**.
- **Algorithm:** Check if any non-key attribute functionally depends on *only a part* of the composite key.
  - *Example:* In `order_item` (PK: `order_id`, `product_id`), an attribute `product_name` depends only on `product_id`, not the whole key.
- **Resolution:** Move the partially-dependent attribute to the entity representing the partial key (move `product_name` to `product` entity).

### 3. Third Normal Form (3NF) Validation
**Goal:** Eliminate Transitive Dependencies.
- **Rule:** Every non-key attribute must provide a fact about the key, the whole key, and *nothing but the key*.
- **Algorithm:** Check for non-key attributes that depend on *other non-key attributes*.
  - *Example:* In `employee` (PK: `emp_id`), we have `dept_id` and `dept_name`. `dept_name` depends on `dept_id`, which is not the primary key.
- **Resolution:** Extract the transitive group into a new entity (`department`), leave `dept_id` in `employee` as a Foreign Key.

## Workflow Steps

### Step 1: Ingest Schema & Dependencies
```javascript
const schemaPath = '.tmp/parsed_schemas/schema_v1.json';
// Ingest schema into the Normalization Engine
```

### Step 2: Execute Normalization Engine
```typescript
// execution/normalize_schema.ts
import { normalizeTo3NF } from './normalize_schema';

const normalizationResult = normalizeTo3NF(parsedSchema, {
  strictMode: false,
  autoDecompose: true
});

if (normalizationResult.issuesFound > 0) {
    console.log(`Detected & resolved ${normalizationResult.issuesFound} normalization anomalies.`);
}
```

### Step 3: Output Decomposed Schema
```typescript
import fs from 'fs/promises';

await fs.writeFile(
  '.tmp/parsed_schemas/normalized_schema.json',
  JSON.stringify(normalizationResult.schema, null, 2)
);
await fs.writeFile(
  '.tmp/normalization_logs/report_v1.md',
  normalizationResult.report
);
```

## Edge Cases & Error Handling

### 1. False Positives (Over-Normalization)
**Problem:** The algorithm detects `zip_code` functionally determining `city` and `state`, and aggressively decomposes an `address` into `zip_code_locations` and `street_addresses`. For simple apps, this is overkill.
**Solution:**
- Implement a whitelist of acceptable transitive dependencies (e.g., standard geo-location fields).
- **Action:** If `strictMode` is false, bypass decomposition for known heuristic groupings like Addresses.

### 2. Lossy Decompositions
**Problem:** A bug in the decomposition logic splits a table but forgets to generate the connecting Foreign Key, permanently destroying the relationship.
**Solution:**
- Run a post-decomposition Lossless Join Test.
- **Action:** Ensure the intersection of the decomposed tables' attributes contains the primary key of at least one of the new tables. If not, revert and log an error.

### 3. Conflicting Primary Keys / Surrogates vs Natural
**Problem:** A decomposition moves an attribute to an entity where no natural primary key exists.
**Solution:**
- **Action:** Auto-generate Surrogate Keys. Whenever a new table is split out during 2NF/3NF processes, automatically inject `{new_entity_name}_id` as an `INTEGER` Primary Key.

### 4. Overlapping Foreign Keys
**Problem:** Moving fields during 3NF creates redundant foreign keys pointing to the same place, creating cycles.
**Solution:**
- **Action:** Run a redundant constraint cleanup pass at the end of the normalization loop. Remove duplicate relationships between the exact same entities if the semantic meaning is identical.

---

## Success Criteria

A successfully normalized schema should guarantee:

✅ **1NF pass:** Zero pluralized or pseudo-array attributes (`tags`, `hobbies`, `image1`, `image2`).
✅ **2NF pass:** No attributes in junction/composite-key tables that describe just one side of the relation.
✅ **3NF pass:** No duplicate data groupings. If `vendor_name` and `vendor_contact` appear in `products`, they are successfully abstracted into a `vendors` table.
✅ **Integrity pass:** No orphaned tables generated during decomposition.

## Testing Strategy

**Test Cases to Run:**
1. **The Array Test:** Provide a schema with `category_1`, `category_2` columns. Ensure it becomes a junction table.
2. **The Receipt Test:** Provide an `order_details` schema with (`order_id`, `product_id`, `product_price`, `customer_name`). Ensure `product_price` goes to `product` (2NF) and `customer_name` drops if `order` already has `customer_id` (3NF transitivity).
3. **Lossless Verification:** Attempt to rejoin all decomposed tables automatically in a test suite.
